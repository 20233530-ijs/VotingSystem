# Part 1: PRD (Product Requirements Document)

## 1.1 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | 블록체인 기반 투표 시스템 (Blockchain Voting System) |
| 네트워크 | Ethereum Sepolia Testnet |
| 접근 방식 | 웹 브라우저 (Chrome + MetaMask 익스텐션) |
| 투표 형태 | 단일 투표 운영 |
| 운영 규모 | 실습용 (최대 동시 접속 5명, 총 투표 30건 미만) |
| 작성일 | 2026-05-11 |

## 1.2 배경 및 목적

기존 온라인 투표 시스템은 중앙 서버에 결과를 저장하므로 데이터 위·변조 가능성이 존재한다. 이더리움 블록체인의 불변성(Immutability)과 투명성을 활용하여 신뢰할 수 있는 투표 시스템을 구축한다.

**목적:**
- 블록체인의 불변성으로 투표 기록(득표수, 참여 여부)의 무결성을 보장한다.
- MetaMask 지갑 주소 기반 본인 확인으로 1지갑 주소당 1표를 강제한다.
- 투표 진행 중 실시간으로 득표 현황을 공개하여 선거 방송과 같은 가시성을 제공한다.
- 블록체인에 꼭 저장할 필요가 없는 메타데이터(후보자명, 사진 등)는 백엔드 서버에서 관리하여 가스비를 최소화한다.

## 1.3 대상 사용자

### 일반 투표자
- MetaMask 지갑을 보유한 사용자 (Sepolia 테스트넷 연결)
- 대학교 블록체인 수업 수강생으로 소규모 접속 환경
- 후보자를 선택하고 투표에 참여하는 주체

### 관리자 (Admin)
- 스마트 컨트랙트 배포자 (Owner 지갑 주소 보유자)
- 후보자 등록, 투표 시작/종료 시간 설정 권한을 단독으로 보유
- 관리자 본인도 일반 투표자로서 투표에 참여할 수 있음

## 1.6 제약사항

- **네트워크:** Ethereum Sepolia Testnet 전용
- **지갑:** Chrome MetaMask 익스텐션 필수
- **투표 단위:** 지갑 주소 1개 = 1표 (관리자 포함, 예외 없음)
- **후보자 등록:** 스마트 컨트랙트 Owner만 가능, 투표 시작 전에만 허용
- **시간 설정:** 스마트 컨트랙트 Owner만 가능, 투표 시작 후 변경 불가
- **투표 취소:** 블록체인 특성상 제출된 투표는 취소·변경 불가
- **규모:** 최대 동시 접속 5명, 총 투표 30건 미만 (실습 환경)

## 1.7 성공 지표

- 중복 투표 방지율 100% (컨트랙트 require 검증)
- 투표 트랜잭션 성공률 95% 이상
- 블록체인 기록과 화면 표시 득표수 일치율 100%

# Part 2: SRS (Software Requirements Specification)

## 2.1 시스템 개요

이더리움 세폴리아 테스트넷 스마트 컨트랙트가 투표의 핵심 로직(득표수, 참여 여부, 권한 제어, 투표 기간)을 담당한다. 백엔드 서버는 블록체인에 저장할 필요 없는 메타데이터(투표 제목, 후보자명, 사진 URL)를 관리한다. 프론트엔드는 두 소스에서 데이터를 결합하여 사용자에게 표시한다.

## 2.2 데이터 저장 책임 분리

| 데이터 | 저장 위치 | 이유 |
|--------|-----------|------|
| 투표 기간 (startTime, endTime) | 스마트 컨트랙트 (온체인) | 컨트랙트가 시간 기반 상태를 직접 enforce해야 함 |
| 후보자 ID (1, 2, 3...) | 스마트 컨트랙트 (온체인) | 득표 집계 키로 사용 |
| 득표수 (candidateId → voteCount) | 스마트 컨트랙트 (온체인) | 무결성 보장 핵심 데이터 |
| 투표 여부 (address → bool) | 스마트 컨트랙트 (온체인) | 중복 투표 방지 핵심 데이터 |
| 투표 제목 | 백엔드 서버 | 투표 로직에 불필요, 가스비 절감 |
| 후보자 이름 | 백엔드 서버 | 투표 로직에 불필요, 가스비 절감 |
| 후보자 사진 URL | 백엔드 서버 | 투표 로직에 불필요, 가스비 절감 |
| 후보자 사진 원본 | 외부 이미지 서버 (URL 참조) | 서버 또는 CDN에서 호스팅 |

## 2.3 기술 스택

| 레이어 | 기술 | 선택 이유 |
|--------|------|-----------|
| 스마트 컨트랙트 언어 | Solidity ^0.8.20 | 최신 LTS 마이너, Sepolia 완전 지원 |
| 컨트랙트 배포 도구 | Remix IDE (browser.remix.ethereum.org) | 별도 설치 없이 브라우저에서 컴파일·배포 가능, 실습 최적 |
| 블록체인 통신 | ethers.js v6 (CDN) | 컨트랙트 호출, Provider 생성, 트랜잭션 대기 |
| 공개 RPC (읽기 전용) | https://rpc.sepolia.org | MetaMask 비연결 상태에서 eth_call 전용 |
| 프론트엔드 | HTML / CSS / Vanilla JavaScript | 실습 규모에 적합, 의존성 최소화, 빌드 도구 불필요 |
| 차트 시각화 | Chart.js v4 (CDN) | 가로 막대·도넛 차트, 직접 스크립트 태그 로드 |
| 백엔드 서버 | Node.js (v20 LTS) + Express | 가볍고 설정이 단순 |
| 데이터베이스 | SQLite (better-sqlite3) | 파일 기반, 소규모 실습에 충분, 설치 단순 |

> **규모 판단 근거:** 동시 접속 최대 5명, 총 투표 30건 미만의 실습 환경이므로 복잡한 인프라가 불필요하다. SQLite 파일 하나로 모든 메타데이터를 충분히 관리할 수 있다.

## 2.4 시스템 아키텍처

```
[사용자 브라우저 (Chrome)]
         │
         ├── REST API (메타데이터 조회/저장) ──────▶ [백엔드 서버 :3000]
         │                                            Node.js + Express
         │                                            SQLite DB
         │
         ├── window.ethereum (MetaMask 연결 시) ───▶ [MetaMask Extension]
         │                                                   │
         │                                           서명된 트랜잭션 / eth_call
         │                                                   │
         │                                                   ▼
         │                                         [Ethereum Sepolia Network]
         │                                                   │
         │                                                   ▼
         └── JsonRpcProvider (MetaMask 비연결 시) ─▶ [Voting Smart Contract]
              rpc.sepolia.org (읽기 전용 eth_call)   (득표수, 투표여부, 기간)
```

**데이터 흐름:**
1. 프론트엔드 → 백엔드 REST API: 후보자 이름·사진·제목 조회
2. 프론트엔드 → 스마트 컨트랙트 (eth_call): 득표수·투표여부·기간 조회
   - MetaMask 연결 시: `new ethers.BrowserProvider(window.ethereum)`
   - MetaMask 비연결 시: `new ethers.JsonRpcProvider(CONFIG.SEPOLIA_RPC)` (읽기 전용)
3. 프론트엔드: 두 소스 데이터를 결합하여 화면 렌더링
4. 투표 실행: 프론트엔드 → MetaMask → 스마트 컨트랙트 트랜잭션 (서명 필수)

## 2.5 백엔드 API 명세

### 공개 API (인증 불필요)

| 메서드 | 경로 | 설명 | 응답 예시 |
|--------|------|------|-----------|
| GET | /api/election | 투표 제목 및 전체 후보자 메타데이터 조회 | `{ title, candidates: [{onChainId, name, imageUrl}] }` |
| GET | /api/candidates | 후보자 목록만 조회 | `[{onChainId, name, imageUrl}]` |
| GET | /api/candidates/:onChainId | 특정 후보자 메타데이터 조회 | `{onChainId, name, imageUrl}` |

### 관리자 API (API Key 인증)

| 메서드 | 경로 | 설명 | 요청 Body |
|--------|------|------|-----------|
| POST | /api/admin/election | 투표 제목 저장 | `{ title }` |
| POST | /api/admin/candidates | 후보자 메타데이터 저장 | `{ onChainId, name, imageUrl }` |

**인증 방식:** 요청 헤더 `X-Admin-Key: <key>` 포함. 키는 백엔드 `.env`의 `ADMIN_API_KEY` 값과 일치해야 한다.

```
요청 예시:
POST /api/admin/candidates
X-Admin-Key: my-secret-key
Content-Type: application/json

{ "onChainId": 1, "name": "홍길동", "imageUrl": "https://..." }
```

> 관리자 API는 스마트 컨트랙트 트랜잭션(addCandidate, setElection)이 성공한 후 순서대로 호출한다. onChainId는 CandidateAdded 이벤트 로그에서 파싱한 값을 사용한다.

### FR-005: 투표 상태 자동 전환

투표 상태는 스마트 컨트랙트의 startTime, endTime과 현재 블록 타임스탬프를 비교하여 자동 결정된다.

| 상태 | 조건 | 화면 동작 |
|------|------|-----------|
| NOT_INITIALIZED (미설정) | !isInitialized | "관리자가 투표를 준비 중입니다." 안내 표시 |
| PENDING (대기 중) | isInitialized && block.timestamp < startTime | 카운트다운 표시, 투표 버튼 비활성 |
| ACTIVE (진행 중) | isInitialized && startTime ≤ block.timestamp < endTime | 실시간 득표 차트 표시, 투표 버튼 활성 |
| ENDED (종료) | isInitialized && block.timestamp ≥ endTime | 최종 결과 차트 표시, 투표 버튼 비활성 |

| 항목 | 내용 |
|------|------|
| ID | FR-005 |
| 상태 결정 기준 | 컨트랙트의 getVoteStatus() 반환값 (0: NOT_INITIALIZED / 1: PENDING / 2: ACTIVE / 3: ENDED) |
| 프론트엔드 갱신 | 30초 주기 폴링 (window.ethereum은 HTTP 기반으로 WebSocket 이벤트 구독 불가. 실시간 이벤트가 필요하다면 Alchemy/Infura WebSocket 공급자를 별도 연결해야 하나, 실습 규모에서는 30초 폴링으로 충분함) |

## 2.8 스마트 컨트랙트 명세

### 2.8.1 데이터 구조

**State Variables**

| 변수명 | 타입 | 설명 |
|--------|------|------|
| owner | address | 컨트랙트 배포자 주소 |
| startTime | uint256 | 투표 시작 시간 (Unix Timestamp) |
| endTime | uint256 | 투표 종료 시간 (Unix Timestamp) |
| candidateCount | uint256 | 등록된 후보자 수 |
| voteCount | mapping(uint256 ⇒ uint256) | candidateId별 득표수 |
| hasVoted | mapping(address ⇒ bool) | 지갑별 투표 완료 여부 |
| voterChoice | mapping(address ⇒ uint256) | 지갑별 선택한 후보자 ID |
| isInitialized | bool | 투표 기간 설정 완료 여부 |

> 후보자 이름, 사진 URL, 투표 제목은 온체인에 저장하지 않는다. 백엔드 서버가 candidateId를 키로 메타데이터를 관리한다.

### 2.8.2 함수 명세

| 함수명 | 접근 제어 | 설명 | 상태 변경 |
|--------|-----------|------|-----------|
| owner() | 공개 | 컨트랙트 Owner 주소 반환 (Solidity public 변수 자동 생성 getter) | No |
| setElection(startTime, endTime) | onlyOwner | 투표 기간 설정 | Yes |
| addCandidate() | onlyOwner | 후보자 ID 슬롯 생성, 새 candidateId 반환 | Yes |
| vote(candidateId) | 모든 주소 (Owner 포함) | 투표 실행 | Yes |
| getVoteStatus() | 공개 | 현재 상태 반환 (0: NOT_INITIALIZED, 1: PENDING, 2: ACTIVE, 3: ENDED) | No |
| getVoteCount(candidateId) | 공개 | 특정 후보자 득표수 반환 | No |
| getAllVoteCounts() | 공개 | 전체 후보자 득표수 배열 반환 | No |
| getTotalVotes() | 공개 | 총 투표수 반환 | No |
| hasVotedAddress(address) | 공개 | 특정 주소의 투표 완료 여부 반환 | No |
| getVoterChoice(address) | 공개 | 특정 주소가 선택한 후보자 ID 반환 | No |
| getElectionPeriod() | 공개 | startTime, endTime 반환 | No |

### 2.8.3 require 조건

| 함수 | 검증 조건 | revert 메시지 |
|------|-----------|---------------|
| setElection | msg.sender == owner | "Only owner" |
| setElection | endTime > startTime | "Invalid time range" |
| setElection | !isInitialized \|\| block.timestamp < startTime | "Election already started" |
| setElection | candidateCount >= 2 | "Need at least 2 candidates" |
| addCandidate | msg.sender == owner | "Only owner" |
| addCandidate | !isInitialized \|\| block.timestamp < startTime | "Election already started" |
| vote | getVoteStatus() == 2 (ACTIVE) | "Election is not active" |
| vote | !hasVoted[msg.sender] | "Already voted" |
| vote | candidateId >= 1 && candidateId <= candidateCount | "Invalid candidate" |

| ElectionSet | startTime, endTime | 투표 기간 설정 시 |
| CandidateAdded | candidateId | 후보자 슬롯 생성 시 |

### 2.10.1 스마트 컨트랙트 배포 절차

```
[6단계] 관리자 초기 설정
  └─ admin.html 접속 (Owner MetaMask 연결)
     → 후보자 2명 이상 등록 → 투표 기간 설정

  └─ 수강생들이 index.html 접속하여 MetaMask로 투표
```

### 2.10.8 이미지 로드 실패 처리

모든 `<img>` 태그에 `onerror` 핸들러를 적용하여 URL이 유효하지 않거나 이미지 서버가 응답하지 않을 때 기본 이미지를 표시한다.

```html
<!-- 예시 -->
<img src="[imageUrl]"
     alt="[name]"
     onerror="this.src='placeholder.png'; this.onerror=null;">
```

`placeholder.png`는 frontend/ 폴더에 포함한다 (회색 사람 실루엣 이미지 권장).

2. 백엔드 POST /api/admin/election { title } 저장 (X-Admin-Key 헤더 포함)
3. MetaMask → setElection(startTime, endTime) 트랜잭션 전송
4. 트랜잭션 receipt 수신 대기 (`tx.wait()`)
5. 성공 토스트 표시 및 설정 상태 갱신

> 서버 저장(2단계)을 컨트랙트 호출(3단계) 앞에 두어, 컨트랙트 트랜잭션이 실패해도 서버 데이터가 남아 재시도 시 일관성을 유지한다. 단 두 단계가 모두 성공해야 설정 완료로 간주한다.

**처리 순서 (후보자 추가):**
1. 입력값 프론트엔드 유효성 검사
2. MetaMask → addCandidate() 트랜잭션 전송
3. 트랜잭션 receipt 수신 대기 (ethers.js: `tx.wait()`)
4. receipt의 CandidateAdded 이벤트 로그에서 candidateId 파싱
5. 백엔드 POST /api/admin/candidates { candidateId, name, imageUrl } 저장
6. 목록 갱신

> ⚠ **데이터 정합성 주의:** 4단계(온체인 트랜잭션)는 성공했으나 5단계(서버 저장)가 실패하면 블록체인에는 ID 슬롯이 있지만 서버에는 메타데이터가 없는 상태가 된다. 이 경우 관리자 대시보드에서 `/api/admin/candidates`를 수동 재호출하거나, 프론트엔드가 재시도 로직을 제공해야 한다.

### 코드 리뷰 체크리스트 (~0.8h)

**스마트 컨트랙트**
- [ ] `getVoteStatus()` 반환값이 NOT_INITIALIZED(0) / PENDING(1) / ACTIVE(2) / ENDED(3) 정확히 분기되는가
- [ ] `setElection` require — `!isInitialized || block.timestamp < startTime` 조건 순서 올바른가
- [ ] `setElection` require — `candidateCount >= 2` 포함되어 있는가
- [ ] `vote` require — `getVoteStatus() == 2` (ACTIVE = 2) 로 되어 있는가
- [ ] `addCandidate()` 반환값 없음 (반환값 대신 `CandidateAdded` 이벤트로 ID 전달)
- [ ] `isInitialized`가 `setElection()` 호출 시 `true`로 변경되는가

- [ ] `X-Admin-Key` 검증이 미들웨어로 분리되어 있는가 (route 함수 내부에 하드코딩되지 않았는가)

- [ ] `.env`의 `ADMIN_API_KEY`가 코드에 직접 노출되지 않았는가
- [ ] DB 초기화가 서버 시작 시 자동으로 이루어지는가 (수동 마이그레이션 불필요)
